## 🔴 阻断性风险 (Must Resolve Before Phase 3)

### R1: 标注数据缺失 ⚠️ BLOCKER
**风险等级**: 🔴 高  
**影响阶段**: Phase 3 (原型验证) → Phase 4  
**发现来源**: delivery-plan.md 中明确标注

**问题描述**:
- 缺少标注数据样例，将直接影响模型训练和效果验证
- Phase 3 的测试数据集验证无数据可用
- 准确率指标无法量化评估

**隐藏假设** (已暴露):
- ❌ "业务方会及时提供标注数据" — 未确认时间点和数据量
- ❌ "现有 query 分布有代表性" — 未经验证

**缓解措施**:
1. **短期**: 立即与业务方确认数据交付时间表（Week 1 必须完成）
2. **备选方案 A**: 使用合成数据 + 主动学习进行冷启动
3. **备选方案 B**: 利用公开意图分类数据集进行迁移学习
4. **降级路径**: 若数据量 <1000条，优先使用规则+关键词方案

**责任方**: strategy-lead (协调业务方)  
**决策点**: Week 2 检查点 — 若数据仍未到位，触发降级路径

---

### R2: 大模型 API 调用预算未确认 ⚠️ BLOCKER
**风险等级**: 🔴 高  
**影响阶段**: Phase 2 → Phase 3

**问题描述**:
- Layer 3 语义增强层依赖大模型推理
- API 调用成本未预算，可能导致技术方案无法落地
- 实时推理延迟和成本权衡未评估

**隐藏假设**:
- ❌ "大模型 API 成本可控" — 未做成本估算
- ❌ "延迟要求可通过技术优化满足" — 未验证

**缓解措施**:
1. **立即行动**: 完成 Phase 1 后立即做成本估算
2. **备选方案**: 使用开源模型本地部署 (Llama/Qwen)
3. **技术方案**: Layer 3 设计为可选模块，支持降级
4. **预算申请**: Week 1 提交预算申请单

**责任方**: strategy-lead (预算申请)  
**决策点**: Phase 2 技术决策评审时确认

## 🟡 执行风险 (May Delay Delivery)

### E1: 需求边界模糊风险
**风险等级**: 🟡 中  
**影响阶段**: Phase 1 → Phase 2

**问题描述**:
- 用户场景收集依赖业务方配合
- 非功能需求（延迟、准确率）阈值可能不明确
- 边界 case 处理策略未定义

**缓解措施**:
1. Week 1 组织需求澄清会
2. 使用 MoSCoW 方法优先级排序
3. 预留需求变更缓冲（Phase 1 延长 2-3 天）

---

### E2: 技术方案返工风险
**风险等级**: 🟡 中  
**影响阶段**: Phase 2 → Phase 3

**问题描述**:
- 候选技术栈可能存在兼容性问题
- POC 验证可能发现不可行点
- 性能基准可能与预期差距大

**缓解措施**:
1. Phase 1 完成关键技术 POC
2. 每个技术决策点准备 Plan B
3. 设置技术评审 checkpoint

---

### E3: 多团队协调风险
**风险等级**: 🟡 中  
**影响阶段**: 全流程

**问题描述**:
- 业务评审时间依赖业务方排期
- 跨团队依赖可能导致阻塞
- 沟通成本被低估

**缓解措施**:
1. 提前预约关键评审会议
2. 建立异步评审机制
3. 设置每周同步例会

---

### E4: 加速模式返工风险
**风险等级**: 🟡 中  
**影响阶段**: Phase 2-3 并行期

**问题描述**:
- 边设计边原型可能导致返工
- 文档同步可能滞后
- 决策链路可能阻塞

**缓解措施**:
1. 增量验证，每个小迭代都有验收
2. 文档先行，至少完成设计大纲再开发
3. 关键决策点设置 gate

## 🔗 依赖风险 (External Dependencies)

### D1: GPU 资源依赖
**风险等级**: 🟡 中  
**依赖方**: 运维团队

**问题描述**:
- 模型训练和推理需要 GPU 资源
- 资源可用性和排期未确认

**缓解措施**:
1. 预留云资源作为 backup

3. Week 1 确认资源排期

---

### D2: 第三方 API 稳定性
**风险等级**: 🟡 中  
**依赖方**: 第三方服务商

**问题描述**:
- 大模型 API 可能存在限流或服务中断
- 响应延迟可能不稳定

**缓解措施**:
1. 多供应商备选（OpenAI, Anthropic, 国内厂商）
2. 本地缓存高频查询结果
3. 设计降级开关

---

### D3: 业务方配合度
**风险等级**: 🟢 低  
**依赖方**: 业务团队

**问题描述**:
- 需求确认、数据提供、评审参与依赖业务方

**缓解措施**:
1. 提前沟通预期和里程碑
2. 准备自助文档减少依赖
3. 设置明确的 SLA 和 deadline

## 🛠️ 技术风险 (Technical Risks)

### T1: Embedding 生成延迟瓶颈
**风险等级**: 🔴 高  
**影响阶段**: Phase 3

**问题描述**:
- Layer 2 嵌入向量生成是关键延迟瓶颈
- P95 < 200ms 要求可能难以满足
- 实时场景下性能挑战大

**缓解措施**:
1. 预计算 + 缓存常见 query embedding
2. 使用轻量化 embedding 模型 (e.g., MiniLM)
3. 批处理请求减少调用次数
4. 考虑近似最近邻检索 (ANN) 替代精确检索

---

### T2: 聚类效果评估标准
**风险等级**: 🟡 中  
**影响阶段**: Phase 2-3

**问题描述**:
- 意图聚类缺乏客观评估指标
- 业务价值和聚类效果映射不清晰
- 可能导致方案难以验证

**缓解措施**:
1. 定义多维评估框架（内部指标+业务指标）
2. 人工评估 + 自动评估结合
3. A/B 测试验证业务价值

---

### T3: 模型可解释性
**风险等级**: 🟢 低  
**影响阶段**: Phase 4 (运维阶段)

**问题描述**:
- 深度学习模型可解释性差
- 错误分类难以调试

**缓解措施**:
1. 保留分类置信度信息
2. 设计可追溯的决策路径
3. 提供人工干预入口

---

## 🔚 结论与建议

### 必须解决 (Before Phase 2)
1. ⚠️ **标注数据**: Week 2 前必须确认数据交付计划，否则触发降级
2. ⚠️ **API 预算**: Week 2 前完成预算评估和申请

1. 提前准备降级方案 (Tier 2/3)
2. Phase 1 完成关键技术 POC 验证
3. 建立数据质量监控机制

### 项目可行性评估
- ✅ 技术方案: 可行，有成熟技术栈支撑
- ⚠️ 数据依赖: 高风险，需立即行动
- ⚠️ 资源依赖: 中风险，需提前确认
- ✅ 时间线: 合理，有缓冲空间

**整体建议**: 项目可以启动 Phase 1，但 Phase 2 前必须解决数据依赖问题。建议设置 Week 2 决策点，届时评估是否继续或降级。
