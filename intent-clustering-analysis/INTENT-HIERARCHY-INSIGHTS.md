## 核心结论一：两种结构的本质差异

### L1/L2/L3 层级结构

```
数学性质：树结构（根-枝-叶）
关系：包含关系 L1 ⊇ L2 ⊇ L3
特点：同一意图空间的不同深度切分

示例：
         L1: 售后服务（领域）
               │
     ┌─────────┼─────────┐
     ↓         ↓         ↓
  L2:物流   L2:退换货  L2:投诉
  （意图）   （意图）   （意图）
     │         │         │
  L3:动作级细分...
```

### RAG 数据聚类结构

```
数学性质：平面集合结构
关系：不相交 簇i ∩ 簇j = ∅
特点：不同意图原型的不相交集合

示例：
  簇1    簇2    簇3    簇4    ...噪声点
 ┌───┐  ┌───┐  ┌───┐  ┌───┐      ○ ○ ○
 │q1│  │q5│  │q9│  │q13│
 │q2│  │q6│  │q10│ │q14│
 │q3│  │q7│  │q11│ │...│
 └───┘  └───┘  └───┘  └───┘
```

### 关系总结

| 特性 | L1/L2/L3 层级 | RAG 聚类簇 |
|-----|--------------|-----------|
| 结构 | 树结构（根-枝-叶） | 平面集合（平行不相交） |
| 关系 | 包含关系 L1 ⊇ L2 ⊇ L3 | 不相交 簇i ∩ 簇j = ∅ |
| 来源 | 人工定义边界 | 数据分布自动发现 |
| 粒度 | 固定层级深度 | 自适应粒度 |
| 目的 | 推理效率（逐层缩小范围） | 意图发现（找到数据中的结构） |

**核心洞察**：层级树 vs 平面集合，这是两种不同的数学结构。RAG 聚类发现的簇，需要经过聚合/拆分/映射，才能变成推理阶段使用的层级框架。

## 核心结论二：语义向量空间是统一坐标系

### 映射机制

所有映射都在语义向量空间中完成：

```
┌─────────────────────────────────────────────────────────────┐
│                 语义向量空间（统一坐标系）                    │
│                                                             │
│   所有query、簇中心、层级边界都在这个空间中表示              │
│                                                             │
│   映射机制：向量距离 = 相似度                                │
│   - Cosine Similarity                                      │
│   - Euclidean Distance                                     │
│   - ANN检索 (HNSW)                                         │
└─────────────────────────────────────────────────────────────┘
```

### 两阶段映射流程

**阶段1：RAG聚类 → 簇中心向量**

```
历史数据 → Embedding → HDBSCAN聚类 → 簇中心向量

簇中心向量 = 簇内所有query向量的均值/中位数
每个簇有一个"代表向量"作为锚点
```

**阶段2：簇中心 → L1/L2/L3层级边界**

```
簇中心向量 → 层次聚类 → 形成层级区域

L1边界 = 聚合相似簇中心的大区域
L2边界 = 中等粒度的簇中心
L3边界 = 最细粒度的簇中心
```

### 在线推理匹配

```
用户query → Embedding(v_query)
         → ANN检索找最近的簇中心
         → 根据距离判断属于哪个层级区域

distance(v_query, v_center_L1) → L1领域归属
distance(v_query, v_center_L2) → L2意图归属
distance(v_query, v_center_L3) → L3动作归属
```

**核心洞察**：语义向量是唯一桥梁 —— RAG聚类产生的簇中心向量，就是在线推理时ANN检索的目标点。两者共享同一个向量空间，所以可以直接匹配。

## 核心结论三：RAG数据与用户query的语义gap

### 问题：语义稀疏导致向量错位

```
RAG数据向量（描述性语义）     用户query向量（意图性语义）
─────────────────────────     ─────────────────────────
内容描述丰富                  意图信号强
专业术语密集                  口语化、模糊
意图信号弱                    缺乏专业术语

向量分布：偏向"内容空间"      向量分布：偏向"意图空间"
```

### 语义偏移示意

```
      "意图空间"维度
            ↑
            │     ★ 用户query（意图信号强）
            │    ╱
            │   ╱  ← 用户query向量集中在意图维度
            │  ╱
            │ ╱
            │╱────────────────────────────→ "内容空间"维度
            │                ★ 文档片段（内容信号强）
            │               ╱
            │              ╱ ← 文档向量集中在内容维度
            │             ╱
            │            ★

两者分布在不同区域 → 语义gap → 匹配效果差
```

### 解决方案：意图对齐

| 方案 | 方法 | 效果 |
|-----|------|------|
| Query重写/扩展 | LLM重写query为专业表达 | 向量更接近文档空间 |
| 文档意图标注 | 添加意图标签到文档头部 | 文档向量增加意图维度 |
| 双塔模型(ColBERT) | Query/Doc分别编码+交互层 | 学习意图匹配而非纯语义 |
| 混合检索 | 向量+关键词融合 | 补充语义gap |
| 意图向量增强 | concat(v_semantic, v_intent) | 显式添加意图维度 |

### 推荐方案：意图导向的RAG预处理

```
Step1: RAG数据意图标注（离线）
  文档 → LLM提取意图标签 → 添加到文档头部
  "物流查询功能支持..." → "[物流查询] 物流查询功能支持..."

Step2: 意图导向Embedding
  不用纯语义模型，用意图训练的模型
  训练数据: query-doc意图匹配pair

Step3: 意图簇中心校准
  簇中心 = mean(v_doc中心, v_query_samples)
  添加真实query样本让簇中心更贴近意图表达

Step4: 在线Query增强
  用户query → 意图识别 → Query扩展
  "查快递" → 扩展"物流查询 订单状态 配送"
```

**核心洞察**：RAG数据向量 ≠ 用户query向量，需要在向量空间中显式添加"意图维度"，或者用意图标注/Query扩展来对齐语义gap。

## 意图粒度映射

### 三层意图粒度定义

| 层级 | 粒度 | 典型类别数量 | 示例 | 映射关系 |
|-----|------|------------|------|---------|
| L1 | 领域级 | 5-20个 | 售后服务、技术支持、产品咨询 | 簇聚合形成的大区域 |
| L2 | 意图级 | 50-200个 | 物流查询、退换货申请、报错诊断 | ≈ 高纯度聚类簇 |
| L3 | 动作级 | 200-1000+ | 查询物流详情(order_id)、提交退货申请 | 簇内细分 |

### Speech Act 与意图框架的关系

```
Speech Act（语言学）          意图框架（业务）
───────────────────────────────────────────────
L_DIRECTIVE（指令类）    →    L2: 物流查询、退换货申请
L_ASSERTIVE（陈述类）    →    L2: 状态汇报、问题描述  
L_COMMISSIVE（承诺类）   →    L2: 确认预约、接受方案
L_EXPRESSIVE（表达类）   →    L2: 感谢反馈、投诉建议
L_DECLARATION（宣告类）   →    L2: 业务流程触发

Speech Act粒度太粗，无法直接用于业务意图分类
但可以作为L1/L2的辅助语言学特征
```

## 实验数据验证

基于 IntentWeight Phase 1B 实验数据：

| 数据集 | 簇数量 | 纯度 | 推断粒度 | 框架设计建议 |
|-------|-------|------|---------|-------------|
| DailyDialog | 90 | 91.87% | ≈ L2意图级 | 可直接用聚类结果作为意图分类标准 |
| BANKING77 | 208 | 79.71% | ≈ L3动作级 | 高纯度簇可直接定义，混合簇需L2聚合 |
| CLINC150 | - | 75.20% | 意图级偏细 | 多领域分散，需要L1/L2/L3全设计 |
| CMID | 129 | 72.08% | 边界模糊 | 医疗场景复杂，需要更多层处理 |
