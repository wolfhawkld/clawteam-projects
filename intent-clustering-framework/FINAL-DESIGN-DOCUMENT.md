# 多层级多维度意图聚类分类框架 - 完整设计方案

> **版本**: v1.0  
> **创建时间**: 2026-03-30  
> **团队**: intent-framework  
> **Strategy Lead**: 最终推荐整合

---

## 执行摘要

本设计方案为 AI 问答系统提供了一个多层级多维度的意图聚类分类框架。经过跨职能团队（系统分析师、交付规划师、风险映射师、决策编辑师）的综合分析，我们推荐采用 **三层架构** 方案：

**核心结论**:
- ✅ **分层设计**: 语义聚类(L1) → 意图分类(L2) → 细粒度路由(L3)
- ✅ **技术选型**: 向量检索+分类模型+规则引擎混合方案
- ✅ **部署策略**: 边缘部署为主，延迟可控、成本可预测
- ⚠️ **关键风险**: 2个阻断性风险需在Phase 2前解决

## 一、问题框架

### 1.1 核心问题

设计多层级多维度意图聚类分类框架，用于AI问答系统，解决三个核心问题：

1. **分层结构**: 应该分哪几层最合理？
2. **技术选型**: 各层使用什么技术进行聚类和意图分类判断？
3. **权衡考量**: 技术选型的性能/成本/可维护性平衡

### 1.2 系统约束

| 约束类型 | 要求 | 来源 |
|---------|-----|------|
| 延迟 | P95 < 200ms, P99 < 500ms | 业务需求 |
| 准确率 | Intent Accuracy > 90% | 业务需求 |
| 可用性 | SLA > 99.9% | 系统标准 |
| 可扩展性 | 支持新增意图类别 | 长期运维 |

## 二、分层设计方案

### 2.1 推荐架构：三层设计

```
┌─────────────────────────────────────────────────────────────────┐
│                     Layer 3: 细粒度路由层                         │
│   (Fine-grained Routing)                                        │
│   职责: 具体任务/动作识别、参数抽取、响应模板匹配                  │
│   技术: NER/槽位填充 + 模板匹配                                  │
│   延迟目标: <50ms                                               │
├─────────────────────────────────────────────────────────────────┤
│                     Layer 2: 意图分类层                          │
│   (Intent Classification)                                       │
│   职责: 业务域意图识别、多标签分类、置信度评估                     │
│   技术: DeBERTa分类器 + 规则增强                                 │
│   延迟目标: <100ms                                              │
├─────────────────────────────────────────────────────────────────┤
│                     Layer 1: 语义聚类层                          │
│   (Semantic Clustering)                                         │
│   职责: 粗粒度语义分组、新意图发现、异常检测                       │
│   技术: 向量检索(HNSW) + HDBSCAN聚类                            │
│   延迟目标: <50ms                                               │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 各层详细设计

#### Layer 1: 语义聚类层

**核心职责**:
- 粗粒度语义分组，快速定位意图范围
- 新意图发现（未被覆盖的query聚类）
- 异常检测（识别无法处理的query）

**技术选型**:

| 组件 | 技术选择 | 选型理由 |
|-----|---------|---------|
| Embedding模型 | bge-m3 / text-embedding-3-small | 中英文平衡，性价比高，延迟15-20ms |
| 向量数据库 | Milvus 2.x / Qdrant 1.x | 支持HNSW索引，ANN检索延迟5-10ms |
| 聚类算法 | HDBSCAN(离线) + KNN(在线) | 自动发现新簇，在线快速匹配 |

**数据流设计**:
```
Query → Embedding Model → Vector DB (ANN Search) → Top-K Clusters → Cluster Assignment
                                     ↓
                              Anomaly Detection (distance threshold)
```

**关键参数**:
- `cluster_distance_threshold`: 0.3-0.5 (越小越严格)
- `anomaly_threshold`: 0.7 (超过此距离视为异常)
- `min_cluster_size`: 100 (HDBSCAN最小簇大小)

#### Layer 2: 意图分类层

**核心职责**:
- 业务域意图精准识别
- 多标签分类（支持层级意图）
- 置信度评估与校准

**技术选型**:

| 组件 | 技术选择 | 选型理由 |
|-----|---------|---------|
| 主分类器 | Finetuned DeBERTa-v3-base | 高准确率，可边缘部署，推理30-40ms |
| 规则引擎 | 关键词规则 + 正则匹配 | 覆盖长尾case，快速迭代，延迟5-10ms |
| 置信度校准 | Temperature Scaling | 提升置信度可靠性 |

**多标签策略**:
```python
# 层级意图支持
intent_hierarchy = {
    "技术支持": ["软件问题", "硬件问题", "账户问题"],
    "产品咨询": ["功能咨询", "价格咨询", "对比咨询"],
    "售后服务": ["退换货", "维修", "投诉"]
}

# 多标签输出
threshold = 0.3  # 置信度阈值
multi_label = [intent for intent, score in predictions if score > threshold]
```

**模型+规则混合策略**:
```
准确率构成：
┌─────────────────────────────────────────────┐
│ ████████████████████ 模型覆盖 (85%)         │
│ ████████████ 规则覆盖 (12%)                 │
│ ██ 未覆盖 (3%)                              │
└─────────────────────────────────────────────┘

规则优先场景：
- 高频明确意图（关键词命中）
- 监管合规要求（必须正确处理）
- 紧急修复case（模型迭代周期外）
```

#### Layer 3: 细粒度路由层

**核心职责**:
- 具体任务/动作识别
- 参数抽取（槽位填充）
- 响应模板匹配

**技术选型**:

| 组件 | 技术选择 | 选型理由 |
|-----|---------|---------|
| 参数抽取 | LUKE / UIE | 通用信息抽取，少样本适应 |
| 槽位填充 | CRF / Biaffine NER | 序列标注经典方案，延迟30-50ms |
| 模板匹配 | Sentence Similarity + 规则 | 混合方案，覆盖广 |

**示例输出**:
```json
{
  "action": "query_logistics",
  "params": {"order_id": "12345"},
  "template": "logistics_query_v1",
  "confidence": 0.95
}
```

## 三、技术选型权衡分析

### 3.1 核心权衡矩阵

| 决策点 | 方案对比 | 推荐 | 权衡理由 |
|-------|---------|------|---------|
| **层数** | 2层 vs 3层 vs 4层 | **3层** | 职责清晰便于独立迭代；延迟可控；复杂度适中 |
| **处理模式** | 全同步 vs 混合 | **同步为主** | 简化运维；关键路径延迟可控 |
| **模型部署** | 边缘 vs 云端 | **边缘为主** | 延迟10-50ms vs 50-200ms；成本可预测 |
| **分类策略** | 纯模型 vs 混合 | **混合** | 规则兜底长尾(12%)；模型提升泛化(85%) |
| **聚类更新** | 实时 vs 批量 | **批量+增量** | 稳定性优先；支持热更新 |

### 3.2 三层 vs 两层架构详细对比

| 维度 | 三层架构 | 两层架构 |
|-----|---------|---------|
| 职责分离 | ✅ 清晰，各层独立演进 | ❌ 意图分类与路由耦合 |
| 新意图发现 | ✅ L1专门负责 | ❌ 需要分类器处理 |
| 总延迟 | ~150-200ms | ~100-150ms |
| 系统复杂度 | 中 | 低 |
| 迭代灵活性 | ✅ 各层可独立优化 | ❌ 牵一发动全身 |
| 准确率目标 | >90% | ~85-88% |

**决策**: 选择三层架构。延迟可通过流水线优化和缓存策略控制在目标范围内。

### 3.3 边缘部署 vs 云端API对比

| 维度 | 边缘部署 | 云端API |
|-----|---------|--------|
| 延迟 | 10-50ms | 50-200ms |
| 成本 | 固定(GPU成本) | 变动(按调用量) |
| 可控性 | 高 | 中 |
| 运维复杂度 | 高 | 低 |
| 数据安全 | 高 | 中 |

**决策**: L1/L2核心分类边缘部署（延迟敏感）；L3复杂抽取可云端（相对低频）。

```
Query Flow Latency Breakdown (目标: <200ms P95)

┌─────────────────────────────────────────────────────────────┐

│  ├─ Embedding: 15-20ms ⚠️ [潜在瓶颈]                        │

├─────────────────────────────────────────────────────────────┤

│  ├─ Preprocessing: 2-5ms                                    │
│  ├─ Model Inference: 30-40ms ⚠️ [潜在瓶颈]                  │
│  │   → 模型蒸馏(Teacher-Student, 延迟降低60%)              │
│  │   → ONNX优化(TensorRT, 延迟降低40%)                      │
│  │   → 早退机制(高置信度提前返回)                           │
│  └─ Rule Engine: 5-10ms                                     │
├─────────────────────────────────────────────────────────────┤
│  L3 Routing (目标: 80ms)                                    │
│  ├─ Slot Filling: 30-50ms ⚠️ [潜在瓶颈]                     │
│  │   → 规则优先(高频模式用正则快速匹配)                     │
│  │   → 级联处理(先规则后模型)                               │
│  ├─ Template Match: 10-20ms                                 │
│  └─ Response Build: 10-20ms                                 │
└─────────────────────────────────────────────────────────────┘

总延迟: ~150-200ms (P95)
```

## 五、风险评估与缓解

### 5.1 风险总览

| 风险类别 | 高风险 | 中风险 | 低风险 |
|---------|-------|-------|-------|
| 执行风险 | 2 | 3 | 1 |
| 依赖风险 | 2 | 2 | 1 |
| 技术风险 | 1 | 3 | 2 |
| 数据风险 | 2 | 1 | 0 |

### 5.2 阻断性风险（必须在Phase 2前解决）

#### R1: 标注数据缺失 🔴 BLOCKER
**问题**: 缺少标注数据样例，直接影响模型训练和效果验证
**缓解措施**:
- 立即与业务方确认数据交付时间表（Week 1）
- 备选：合成数据+主动学习冷启动
- 降级：若数据量<1000条，使用规则+关键词方案

#### R2: 大模型API预算未确认 🔴 BLOCKER
**问题**: Layer 3依赖大模型推理，成本未预算
**缓解措施**:
- Phase 1后立即成本估算
- 备选：开源模型本地部署(Llama/Qwen)
- L3设计为可选模块，支持降级

### 5.3 隐藏假设清单

| # | 假设 | 验证状态 | 风险 |
|---|------|---------|------|
| 1 | 业务方及时提供标注数据 | ❌ 未验证 | 🔴 高 |
| 2 | 大模型API成本可控 | ❌ 未验证 | 🔴 高 |
| 3 | GPU资源可用 | ❌ 未验证 | 🟡 中 |
| 4 | 延迟要求可通过优化满足 | ❌ 未验证 | 🔴 高 |
| 5 | 现有query分布有代表性 | ❌ 未验证 | 🟡 中 |

### 5.4 降级方案链

```
Tier 1 (完整方案): 4层+大模型
  ↓ 准确率>92%, 延迟~250-300ms
  
Tier 2 (标准方案): 3层+轻量化模型 ← 推荐
  ↓ 准确率>90%, 延迟~150-200ms
  
Tier 3 (最小方案): 2层+规则+聚类
  ↓ 准确率>75%, 延迟~50-100ms
```

## 八、最终推荐

### 推荐方案

| 决策 | 选择 | 理由 |
|-----|------|------|
| **分层结构** | 三层架构 | 职责清晰，独立迭代，延迟可控 |
| **L1技术** | 向量检索+HDBSCAN聚类 | 新意图发现能力强 |
| **L2技术** | DeBERTa分类器+规则增强 | 准确率高，覆盖长尾 |
| **L3技术** | NER槽位填充+模板匹配 | 精细化处理 |
| **部署策略** | 边缘部署为主 | 延迟可控，成本可预测 |
| **交付周期** | 8周标准/6周加速 | 有缓冲，可降级 |

### 拒绝的替代方案

| 方案 | 拒绝理由 |
|-----|---------|
| 两层架构 | 职责耦合，难以独立迭代，准确率目标难达成 |
| 四层架构 | 延迟超标，复杂度过高，运维负担大 |
| 纯云端API部署 | 延迟不可控，成本变动风险高 |
| 纯模型方案（无规则） | 长尾case覆盖率低，迭代周期长 |

### 关键风险

1. ⚠️ **标注数据**: Week 2前必须确认，否则触发降级
2. ⚠️ **API预算**: Week 2前完成评估和申请
