## 一、问题背景

用户需要跨多个客户的文档数据进行查询和总结，例如：
- "对比所有客户的合同条款差异"
- "总结各客户的投诉处理模式"
- "分析跨客户的项目交付趋势"

这需要一个能支持多维度、跨租户检索的架构设计。

---

## 二、方案详细对比

### 方案1：扩展树状Tag逻辑

#### 架构设计

```
Tag层级结构：
├── 客户维度 (client:xxx)
│   ├── 客户A
│   └── 客户B
├── 业务域维度 (domain:xxx)
│   ├── 合同
│   ├── 投诉
│   └── 项目交付
└── 文档类型维度 (type:xxx)
    ├── PDF
    ├── Word
    └── 邮件
```

#### 查询示例
```
# 跨客户查询
tag_filter: "domain:合同" → 返回所有客户的合同文档

# 组合查询
tag_filter: "domain:投诉 AND type:邮件" → 所有客户的投诉邮件
```

#### 实现路径

1. **Tag Schema 扩展**
   - 定义标准维度：`client`, `domain`, `type`, `date`, `status`
   - 支持自定义维度扩展
   - Tag 值规范化（枚举 vs 自由文本）

2. **索引优化**
   - 多维度联合索引
   - Tag 倒排索引优化
   - 缓存热点 Tag 组合

3. **查询层**
   - Tag 组合 DSL
   - 智能推荐相关 Tag
   - 权限控制（租户隔离）

#### 技术栈建议
- **存储**: Elasticsearch / OpenSearch（原生支持多字段过滤）
- **Tag 管理**: 独立 Tag Service
- **查询层**: GraphQL 或 REST API

### 方案2：引入知识图谱RAG

#### 架构设计

```
实体类型：
- Client（客户）
- Document（文档）
- Domain（业务域）
- Topic（主题）
- Entity（命名实体：人、公司、地点等）

关系类型：
- BELONGS_TO（属于）
- REFERENCES（引用）
- SIMILAR_TO（相似）
- RELATED_TO（相关）
- DERIVED_FROM（衍生自）
```

#### 查询流程

```
用户查询："对比各客户的合同违约条款"
    ↓
意图解析 → 识别实体：客户、合同、违约条款
    ↓
图谱查询 → 
    (Document)-[BELONGS_TO]->(Domain:合同)
    (Document)-[CONTAINS]->(Topic:违约条款)
    (Document)-[BELONGS_TO]->(Client)
    ↓
图遍历 + 向量检索混合
    ↓
LLM 总结对比结果
```

#### 实现路径

1. **知识图谱构建**
   - 实体抽取（NER）：从文档中提取实体
   - 关系抽取：识别实体间关系
   - 图谱存储：Neo4j / NebulaGraph / ArangoDB

2. **混合检索**
   - 图谱查询（结构化关系）
   - 向量检索（语义相似）
   - 结果融合与排序

3. **知识推理**
   - 隐式关系发现
   - 语义路径查询
   - 图谱演化更新

#### 技术栈建议
- **图谱存储**: Neo4j（成熟）/ NebulaGraph（高性能分布式）
- **向量库**: Milvus / Weaviate
- **LLM**: 用于实体/关系抽取 + 答案生成
- **编排**: LangGraph / Haystack

## 三、多维度评估

| 维度 | 方案1：树状Tag | 方案2：知识图谱RAG |
|------|---------------|-------------------|
| **实现成本** | ⭐⭐⭐⭐⭐ 低 | ⭐⭐ 高 |
| **查询效率** | ⭐⭐⭐⭐⭐ 高 | ⭐⭐⭐ 中 |
| **扩展性** | ⭐⭐⭐ 中 | ⭐⭐⭐⭐⭐ 高 |
| **灵活性** | ⭐⭐ 低 | ⭐⭐⭐⭐⭐ 高 |
| **维护成本** | ⭐⭐⭐⭐ 低 | ⭐⭐ 高 |
| **关系表达** | ⭐⭐ 弱 | ⭐⭐⭐⭐⭐ 强 |
| **隐性发现** | ⭐ 无 | ⭐⭐⭐⭐⭐ 强 |

### 详细分析

#### 实现成本
- **Tag方案**: 2-4周可上线，复用现有索引基础设施
- **图谱方案**: 2-3个月，需要构建图谱、训练抽取模型、设计混合检索

#### 查询效率
- **Tag方案**: 毫秒级，直接索引过滤
- **图谱方案**: 秒级，需要图遍历 + 向量计算 + 结果融合

#### 扩展性
- **Tag方案**: 新增维度需要修改Schema，历史数据需重新打标
- **图谱方案**: 新增实体类型/关系类型灵活，无需重建

#### 适用场景
- **Tag方案**: 
  - 查询模式固定且已知
  - 维度较少（<10个）
  - 用户明确知道要查什么
  
- **图谱方案**:
  - 需要发现隐式关系
  - 查询模式复杂多变
  - 需要语义推理

## 四、推荐方案

### 短期（0-6个月）：方案1 + 增强

**推荐采用方案1（树状Tag），但增加以下增强**：

1. **Tag 层次化**
   ```
   domain:合同 > domain:合同.条款 > domain:合同.条款.违约
   ```
   支持层级聚合查询

2. **Tag 自动推荐**
   - 基于用户查询历史推荐相关Tag
   - 基于文档内容自动推荐Tag

3. **Tag 关联矩阵**
   - 预计算Tag共现关系
   - 支持弱关联查询

4. **增量迁移路径**
   - 为方案2预留数据结构
   - Tag 映射到图谱实体（为未来迁移做准备）

### 长期（6-18个月）：渐进式引入图谱

**当出现以下需求时，启动方案2**：

1. 用户需要发现"不知道该问什么"的隐式关系
2. Tag 维度爆炸（>20个维度）
3. 需要"为什么相关"的可解释性
4. 需要跨文档的推理问答

**渐进路径**：
```
阶段1：Tag + 向量检索（当前）
    ↓
阶段2：Tag + 轻量实体识别（3个月）
    ↓
阶段3：Tag + 实体图谱（6个月）
    ↓
阶段4：完整知识图谱RAG（12-18个月）
```

## 六、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Tag 膨胀 | 查询效率下降 | Tag 归一化 + 层级约束 |
| 实体抽取不准 | 图谱质量差 | 人机协同标注 + 持续优化 |
| 跨客户数据泄露 | 合规风险 | 严格租户隔离 + 权限审计 |
| 查询响应慢 | 用户体验差 | 异步查询 + 结果缓存 |

---

## 七、结论

**最终推荐：先Tag后图谱的渐进式路径**

1. **立即**：实施方案1（树状Tag），4-6周可上线
2. **并行**：启动图谱能力建设，为Phase 2做准备
3. **演进**：根据实际需求，渐进引入图谱能力

**理由**：
- 满足当前核心需求（跨客户查询）
- 实现成本低，快速验证价值
- 为未来发展预留扩展空间
- 风险可控，渐进迭代
